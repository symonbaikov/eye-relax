# LookAway — Правила разработки

> Этот документ — источник истины для всех контрибьюторов. Прочитай перед первым коммитом.

---

## 1. Философия проекта

LookAway — это утилита-компаньон, а не инструмент контроля. Каждое решение проходит через фильтр: «пользователь будет рад этому или раздражён?». Если ответ неоднозначен — не добавляй.

Приложение должно быть незаметным. Оно не должно потреблять заметные ресурсы, не должно мешать работе, не должно требовать внимания когда в нём нет необходимости. Tray-иконка и тихий overlay — единственные точки контакта.

---

## 2. Архитектурные правила

### 2.1 Три слоя — без исключений

```
Rust Backend → Tauri IPC → React UI → SQLite Storage
```

- Бизнес-логика живёт только в Rust. React — тонкий клиент.
- UI не вызывает SQLite напрямую. Все данные проходят через IPC commands.
- Модули backend не импортируют друг друга напрямую. Коммуникация — через `EventBus`.

### 2.2 Зависимости направлены внутрь

```
UI → Commands → Core Modules → Traits ← Adapters
```

Core-модули (`Scheduler`, `ActivityTracker`, `StatsAggregator`) зависят от трейтов (`StoragePort`, `ActivitySource`, `NotificationPort`), а не от конкретных реализаций. Конкретные адаптеры (`SqliteStorage`, `X11IdleSource`) инжектируются в `main.rs`.

### 2.3 Один модуль — одна ответственность

Перед добавлением метода в модуль задай вопрос: «это ответственность этого модуля?». Если `Scheduler` начинает писать в базу — что-то пошло не так. Scheduler эмитит событие, `StatsAggregator` его ловит и пишет.

### 2.4 Новый функционал — через расширение

Не модифицируй существующие трейты без крайней необходимости. Новый тип перерыва — новая реализация `BreakStrategy`. Новый источник idle — новая реализация `ActivitySource`. Новый бэкенд хранения — новая реализация `StoragePort`.

## 3. Rust backend

### 3.1 Обработка ошибок

- Используй `Result<T, AppError>` везде. Никаких `unwrap()` в продакшн-коде.
- `unwrap()` допустим только в тестах и в начальной инициализации (`main.rs`), где паника — корректное поведение.
- Определи единый `AppError` enum с вариантами: `Storage`, `Platform`, `Config`, `Notification`. Реализуй `From<T>` для каждого внутреннего типа ошибок.
- В IPC commands возвращай человекочитаемые сообщения через `serde`. Фронт не должен парсить Rust-ошибки.

### 3.2 Асинхронность

- Runtime: `tokio` с `rt-multi-thread` feature.
- Фоновые задачи (`ActivityTracker` polling, `Scheduler` timer) — через `tokio::spawn`. Храни `JoinHandle` для graceful shutdown.
- Таймеры — `tokio::time::interval`, не `std::thread::sleep`.
- Каналы — `tokio::sync::broadcast` для EventBus, `tokio::sync::watch` для состояния Scheduler.

### 3.3 Конкурентность

- Разделяемое состояние — через `Arc<Mutex<T>>` или `Arc<RwLock<T>>`.
- Предпочитай `RwLock` для данных с частым чтением (конфиг, состояние).
- Держи критические секции минимальными. Никакого I/O под локом.
- `EventBus.emit()` — неблокирующий. Если буфер полон — событие дропается с логом `warn!`.

### 3.4 Платформенный код

- Весь платформозависимый код — в `src/platform/`.
- Каждый адаптер — отдельный файл (`x11.rs`, `wayland.rs`).
- Выбор адаптера — один раз в `main.rs` через `detect_session_type()`.
- Никаких `#[cfg(target_os)]` вне `platform/`. Всё остальное — кроссплатформенное.

### 3.5 Логирование

- Используй `tracing` crate: `info!`, `warn!`, `error!`, `debug!`, `trace!`.
- Каждый переход FSM — `info!`: `"State transition: {:?} → {:?}"`.
- Каждая ошибка — `error!` с контекстом: `"Failed to save config: {err}"`.
- Пользовательские действия (skip, snooze, pause) — `info!`.
- Polling и тики — `trace!` (не засоряй лог).
- Никаких `println!` и `eprintln!`.

## 4. React frontend

### 4.1 Компоненты

- Функциональные компоненты + хуки. Никаких классовых компонентов.
- Один компонент — один файл. Имя файла = имя компонента: `OverlayWindow.tsx`.
- Компоненты не содержат бизнес-логику. Только отображение и вызов IPC.
- Props типизированы через `interface`, не `type` (для консистентности).

### 4.2 State management

- Zustand — единственный state manager. Никаких Context API для глобального состояния.
- Каждый стор — отдельный файл в `stores/`.
- Компоненты подписываются на конкретные поля через селекторы: `useSchedulerStore(s => s.remaining)`. Не подписывайся на весь стор.
- Сторы не вызывают друг друга напрямую.

### 4.3 Стилизация

- Tailwind CSS — единственный способ стилизации. Никаких CSS-модулей, styled-components, inline styles.
- Исключение: SVG-элементы (прогресс-бар в overlay) могут использовать inline `style` для динамических значений.
- Цветовая палитра определена в `tailwind.config.ts`. Не используй произвольные hex-значения в классах.
- Тёмная тема — через `dark:` модификатор Tailwind. Не дублируй компоненты.

### 4.4 Анимации

- Framer Motion — единственная библиотека анимаций.
- Анимации overlay: `opacity` и `scale` с `duration: 0.3`. Не анимируй `width`, `height`, `top`, `left`.
- Обязательно: `prefers-reduced-motion` — отключай анимации если пользователь просил.
- Анимации не должны блокировать взаимодействие. Кнопки Skip / Snooze кликабельны даже во время анимации появления.

### 4.5 IPC-вызовы

- Все `invoke()` вызовы вынесены в `src/lib/ipc.ts` как типизированные функции.
- Компоненты не вызывают `invoke()` напрямую — только через `ipc.ts`.
- Обработка ошибок IPC — `try/catch` с fallback в UI (toast или inline-сообщение). Не молчаливые ошибки.

## 5. Именование

### 5.1 Rust

- Модули, функции, переменные: `snake_case`
- Типы, трейты, enum'ы: `PascalCase`
- Константы: `SCREAMING_SNAKE_CASE`
- Трейты-порты: суффикс `Port` — `StoragePort`, `NotificationPort`
- Реализации: описательное имя — `SqliteStorage`, `DbusNotifier`, `X11IdleSource`
- Enum-варианты событий: глагол + существительное — `BreakDue`, `UserReturned`, `ConfigUpdated`

### 5.2 TypeScript

- Компоненты: `PascalCase` — `OverlayWindow.tsx`
- Хуки: `camelCase` с префиксом `use` — `useTauriEvents.ts`
- Сторы: `camelCase` с префиксом `use` — `useSchedulerStore.ts`
- Интерфейсы: `PascalCase`, без префикса `I` — `SchedulerStore`, не `ISchedulerStore`
- IPC-функции: `camelCase` — `getState()`, `skipBreak()`
- Файлы утилит: `camelCase` — `ipc.ts`, `formatTime.ts`

### 5.3 Git

- Ветки: `feature/scheduler-fsm`, `fix/overlay-double-open`, `refactor/storage-trait`
- Коммиты: conventional commits — `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`
- Тело коммита на английском. Короткое описание, императив: `feat: add snooze duration config`

## 6. Идемпотентность

Каждая операция в системе должна быть идемпотентной. Повторный вызов = тот же результат. Это не рекомендация — это требование.

### Чеклист при добавлении новой операции

- [ ] Повторный вызов из того же состояния — no-op или перезапись?
- [ ] Запись в БД использует `INSERT OR REPLACE` / `ON CONFLICT` по уникальному ключу?
- [ ] Создание окна/ресурса проверяет существование перед созданием?
- [ ] FSM-переход валидирует текущее состояние перед выполнением?
- [ ] Эмит события не вызывает каскадный повторный эмит?

## 7. Storage

### 7.1 Миграции

- Каждая миграция — отдельная функция `migrate_v{N}_to_v{N+1}()`.
- Все `CREATE TABLE` — с `IF NOT EXISTS`.
- Все `ALTER TABLE ADD COLUMN` — с предварительной проверкой через `PRAGMA table_info`.
- Миграции применяются последовательно при старте. Версия — `PRAGMA user_version`.
- Миграция никогда не удаляет данные. Только добавление столбцов и таблиц.

### 7.2 Правила записи

- Все записи имеют UUID как primary key. Генерируется на стороне Rust через `uuid::Uuid::new_v4()`.
- Даты — ISO 8601 строки (`chrono::Utc::now().to_rfc3339()`).
- Числовые значения — в секундах (не минуты, не миллисекунды).
- `StatsAggregator` пишет в БД батчами: аккумулирует `SessionTick` события и flush каждые 60 секунд.

### 7.3 WAL-режим

SQLite открывается с `PRAGMA journal_mode=WAL`. Это позволяет одновременное чтение и запись без блокировок. Не меняй journal mode.

## 8. Тестирование

### 8.1 Покрытие

- Каждый трейт имеет мок-реализацию в `#[cfg(test)]` модуле.
- Каждый переход FSM покрыт тестом.
- Каждая граница валидации конфига покрыта тестом (min, max, min-1, max+1).
- Идемпотентность каждой операции покрыта тестом (вызов → повторный вызов → assert same result).

### 8.2 Организация

- Unit-тесты — в том же файле: `#[cfg(test)] mod tests { ... }`.
- Интеграционные тесты — в `tests/` каталоге.
- Имена тестов: `{модуль}_{сценарий}_{ожидание}` — `scheduler_double_start_noop`, `config_out_of_range_rejected`.

### 8.3 Async-тесты

- Используй `#[tokio::test]` для async тестов.
- Для детерминистичного времени: `tokio::time::pause()` + `tokio::time::advance()`.
- Никаких `std::thread::sleep` в тестах — только `tokio::time`.

### 8.4 Фронтенд-тесты

- Компоненты: Vitest + Testing Library. Тестируй поведение, не реализацию.
- Сторы: отдельные unit-тесты без рендера. Проверяй переходы состояний.
- IPC-вызовы: мокай `@tauri-apps/api/core` через `vi.mock()`.

## 9. Конфигурация

### 9.1 Валидация

Каждое поле `AppConfig` имеет жёсткие границы. Значение вне границ — `Result::Err`. Не кламп, не тихий fallback на дефолт. Пользователь должен знать что значение невалидно.

### 9.2 Дефолты

Дефолтные значения определены через `impl Default for AppConfig`. Единственный источник правды. Не дублируй дефолты в UI, в SQL, в документации — ссылайся на `Default`.

### 9.3 Hot reload

Изменение конфига не требует перезапуска. `ConfigManager` эмитит `Event::ConfigUpdated` → `Scheduler` пересоздаёт таймер с новым интервалом → UI обновляет отображение. Цепочка полностью реактивная.

## 10. UX-правила

### 10.1 Overlay

- Overlay появляется плавно (300ms fade-in), не резко.
- Overlay не перекрывает весь экран. Окно 400×300px по центру.
- Overlay не блокирует ввод в другие окна (не modal).
- Кнопки Skip и Snooze доступны сразу, без задержки.
- Если пользователь в полноэкранном приложении — показать нотификацию вместо overlay.

### 10.2 Звук

- Звук перерыва — тихий, мягкий, однократный. Не повторяющийся.
- Звук отключается одним тогглом. Не требует перезапуска.
- По умолчанию звук включён.

### 10.3 Tray

- Tray-меню показывает текущий статус и оставшееся время.
- Текст статуса обновляется каждую секунду только когда меню открыто (не тратить ресурсы).
- Иконка меняется при смене состояния — визуальная обратная связь без открытия меню.

### 10.4 Уведомления

- Максимум одно уведомление одновременно. Новое заменяет старое.
- Уведомления — дополнение к overlay, не замена. Если overlay показан — уведомление не нужно.
- Текст уведомления — дружелюбный, без императива. «Время для перерыва» вместо «Сделайте перерыв!».

## 11. Производительность

### 11.1 Ресурсы

- Потребление RAM: целевой показатель < 50 MB в фоне.
- CPU в idle: 0%. Никаких busy loops. Только event-driven и interval polling.
- Polling ActivityTracker — не чаще 30 секунд. Этого достаточно для idle detection.

### 11.2 SQLite

- Batch-запись статистики: flush каждые 60 секунд, не каждый тик.
- Read-операции для UI (статистика, конфиг) — без кеша, напрямую из SQLite. WAL-режим делает чтение неблокирующим.
- Никаких `SELECT *` — запрашивай только нужные колонки.

### 11.3 UI

- Компоненты подписываются на минимальные срезы стора через селекторы.
- `break-tick` обновляет только `remaining` в сторе, не весь объект состояния.
- SVG прогресс-бар в overlay — CSS-анимация `transition`, не React-ререндер каждую секунду.

## 12. Безопасность

- Никаких shell-инъекций. Платформенные команды — только через библиотечные API (`x11rb`, `wayland-client`, `zbus`), не через `Command::new("sh")`.
- SQLite-запросы — только параметризованные (`?`). Никакого `format!()` для SQL.
- IPC commands валидируют входные данные на стороне Rust. Фронт не trusted.
- Данные пользователя (статистика, настройки) хранятся локально. Никакой телеметрии, никаких внешних запросов.

## 13. Релиз и сборка

### 13.1 Формат

- Основной: AppImage — один файл, без зависимостей, работает на любом дистрибутиве.
- Дополнительно: `.deb` для Ubuntu/Debian, Flatpak для песочницы.
- Тестируй на: Ubuntu 24.04 (GNOME/Wayland), Fedora 40 (GNOME/Wayland), Arch (KDE/X11).

### 13.2 CI

- Каждый PR: `cargo clippy -- -D warnings` + `cargo test` + `npm run lint` + `npm run typecheck`.
- Clippy без предупреждений. Если clippy ругается — исправь, не подавляй (`#[allow]` только с комментарием почему).
- Форматирование: `cargo fmt` + `prettier` для TypeScript. Проверяется в CI.

### 13.3 Версионирование

- Semver: `MAJOR.MINOR.PATCH`.
- `MAJOR`: сломанная обратная совместимость конфига или IPC.
- `MINOR`: новая фича (новый тип перерыва, новая страница UI).
- `PATCH`: баг-фиксы, тюнинг, обновление зависимостей.
- Версия хранится в `Cargo.toml` и `package.json`. Синхронизируй при релизе.
